# Plan: api — FastAPI HTTP Layer

**Status:** PLANNED
**Written:** April 2026

---

## Goal

Implement the `api` package: a FastAPI application that exposes the game over
HTTP.  This is a thin adapter layer — it loads snapshots from the store, calls
the game engine, updates application state, saves the result, and returns a
response.  All game logic stays in the engine modules; the API layer orchestrates
and translates.

After this phase the game is network-accessible and end-to-end testable with
plain HTTP calls.  A browser, curl, or a future PWA frontend can drive the full
player lifecycle.

## Identity Scheme

Player identity is a UUID supplied as the `Avatar-Id` request header.  No
authentication in Phase 8; the header value is trusted as-is.

All state-changing endpoints extract this header via a FastAPI dependency
(`get_avatar_id`).  The `GET /avatar/{id}` endpoint reads the ID from the
URL path instead.

## Files

### `src/sender_frenz/api/_helpers.py`

Pure helper functions with no FastAPI coupling — testable in isolation.

```python
def update_sustained_since(
    avatar: Avatar,
    sustained_since: Timestamp | None,
    now: Timestamp,
    pace: GamePace,
) -> Timestamp | None:
    """Recompute sustained_since after any state change.

    Contract (from Phase 7 persistence plan):
    - combined_health(avatar) >= threshold AND sustained_since is None
      → set sustained_since = now
    - combined_health(avatar) < threshold → None
    - Otherwise → carry sustained_since forward unchanged
    """
    config = LevelConfig.from_pace(pace)
    if combined_health(avatar) >= config.threshold:
        return sustained_since if sustained_since is not None else now
    return None


def make_snapshot(avatar_id: AvatarId, now: Timestamp) -> GameSnapshot:
    """Create a brand-new GameSnapshot for a first-time avatar.

    Called when the store has no existing snapshot for avatar_id.
    Creates avatar and room via the character_builder / space_builder
    factories, sets last_tick = now, sustained_since = None.
    """
```

### `src/sender_frenz/api/events.py`

In-memory event bus for SSE delivery.  One asyncio.Queue per connected client
registered for an avatar ID.

```python
class EventBus:
    """Fan-out event bus: avatar_id → list of subscriber queues.

    Not thread-safe.  All FastAPI route handlers run in the same asyncio
    event loop; no locking is required.
    """

    def subscribe(self, avatar_id: AvatarId) -> asyncio.Queue[GameEvent | None]:
        """Register a new subscriber and return its queue.

        A sentinel value of None signals the generator to close.
        """

    def unsubscribe(self, avatar_id: AvatarId, queue: asyncio.Queue[..]) -> None:
        """Remove a subscriber queue."""

    def publish(self, avatar_id: AvatarId, events: Iterable[GameEvent]) -> None:
        """Push events to all queues registered for avatar_id."""
```

### `src/sender_frenz/api/routes/session.py`

**`POST /session/open`**

1. Extract `Avatar-Id` header → `AvatarId`.
2. Load snapshot from store; if `None`, call `make_snapshot(avatar_id, now)`.
3. Call `open_session(avatar, history, sustained_since, now, pace, quip_caller)`.
4. Compute updated `sustained_since` via `update_sustained_since`.
5. Build new `GameSnapshot(avatar=session.avatar, room=snapshot.room,
   history=snapshot.history, sustained_since=..., last_tick=now)`.
6. Save to store.
7. Publish `session.events` to `bus`.
8. Return `SessionResponse`.

**`GET /avatar/{id}`**

1. Parse `{id}` as UUID → `AvatarId`.
2. Load snapshot from store.
3. Return 404 if `None`.
4. Return `AvatarResponse` without modifying state.

### `src/sender_frenz/api/routes/action.py`

**`POST /action/{kind}`** where `kind` ∈ `{"feed", "clean", "visit", "gift", "chat"}`

1. Extract `Avatar-Id` header → `AvatarId`.
2. Load snapshot (or bootstrap if absent).
3. Dispatch:
   - `feed` → `required_maintenance.actions.feed(avatar, now, quip_caller)`
   - `clean` → `required_maintenance.actions.clean(avatar, now, quip_caller)`
   - `visit` / `gift` / `chat` → `social_maintenance.interactions.interact(
       avatar, InteractionKind(kind), now, quip_caller)`
4. Compute updated `sustained_since`.
5. Build new snapshot with updated avatar; save; publish events (empty list
   for action — the event stream carries game-loop events only, not action
   confirmations).
6. Return `ActionResponse`.

Raise `HTTPException(422)` for unknown `kind`.

### `src/sender_frenz/api/routes/level_up.py`

**`POST /level-up`**

Request body: `LevelUpRequest { skin_slug, room_slug }`.

1. Extract `Avatar-Id` header → `AvatarId`.
2. Load snapshot; 404 if absent.
3. Check `is_level_up_available`; 409 if not available.
4. Call `apply_level_up(avatar, room, skin_slug, room_slug, SKIN_CATALOG, ROOM_CATALOG)`.
   Catch `ValueError` → 422.
5. Set `sustained_since = None` (streak resets on level-up).
6. Save snapshot; return `LevelUpResponse` with available options at new level.

### `src/sender_frenz/api/routes/stream.py`

**`GET /events/{avatar_id}`**

Server-Sent Events stream.  Long-lived HTTP connection; events are pushed
as they occur via the `EventBus`.

```
Content-Type: text/event-stream
Cache-Control: no-cache
```

Each event line:
```
data: {"kind": "hunger_warning", "timestamp": 1744000000.0}\n\n
```

Implementation:

```python
async def _sse_generator(
    avatar_id: AvatarId,
    bus: EventBus,
) -> AsyncIterator[str]:
    queue = bus.subscribe(avatar_id)
    try:
        while True:
            event = await queue.get()
            if event is None:
                break
            payload = json.dumps({"kind": event.kind, "timestamp": event.timestamp})
            yield f"data: {payload}\n\n"
    finally:
        bus.unsubscribe(avatar_id, queue)


@router.get("/events/{avatar_id}")
async def events_stream(
    avatar_id: AvatarId,
    bus: Annotated[EventBus, Depends(get_bus)],
) -> StreamingResponse:
    return StreamingResponse(
        _sse_generator(avatar_id, bus),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache"},
    )
```

The `_sse_generator` body is marked `# pragma: no cover` — it requires a
live SSE connection to test, which is out of scope for unit tests.
Integration testing the SSE stream is deferred to Phase 9.

## `sustained_since` Update Logic

The full contract, replicated from the Phase 7 plan for implementation reference:

| Condition | Action |
|---|---|
| `combined_health >= threshold` and `sustained_since is None` | `sustained_since = now` |
| `combined_health >= threshold` and `sustained_since is not None` | carry forward unchanged |
| `combined_health < threshold` | `sustained_since = None` |
| After `apply_level_up` (regardless of health) | `sustained_since = None` |

This logic lives exclusively in `_helpers.update_sustained_since`, called
after every state-changing endpoint.  Level-up explicitly overrides with
`None` after calling `apply_level_up`.

## Bootstrap: New Avatar Flow

When `store.load(avatar_id)` returns `None`, `make_snapshot` creates:

1. A fresh `Avatar` via `character_builder.avatar.new_avatar(avatar_id, now)`.
2. A fresh `Room` via `space_builder.room.new_room(room_id, avatar_id)` where
   `room_id` is a new `RoomId(uuid4())`.
3. An empty `InteractionHistory` via `social_maintenance.history.create_history()`.
4. `GameSnapshot(avatar=..., room=..., history=..., sustained_since=None, last_tick=now)`.

The bootstrapped snapshot is used for the current request but is NOT saved to
the store until the request produces a state change.  `GET /avatar/{id}` returns
404 for an avatar that has never opened a session.

### `tests/api/test_helpers.py`

- `update_sustained_since` returns `now` when health ≥ threshold and
  `sustained_since` is `None`.
- Returns `sustained_since` unchanged when health ≥ threshold and already set.
- Returns `None` when health < threshold.
- `make_snapshot` returns a `GameSnapshot` with the correct `avatar_id`,
  `sustained_since=None`, and `last_tick=now`.
- `make_snapshot` returns a snapshot that satisfies `isinstance(store, StoreProtocol)`
  after saving (sanity check for integration).

### `tests/api/test_events.py`

- `subscribe` returns a `Queue`.
- `publish` enqueues events to all subscribers for an avatar ID.
- `publish` does not enqueue to subscribers for a different avatar ID.
- `unsubscribe` removes the queue; subsequent publishes are not delivered.
- Subscribing twice returns two independent queues; both receive events.

### `tests/api/test_session.py`

**`POST /session/open`**

- Returns 422 when `Avatar-Id` header is missing.
- Returns 422 when `Avatar-Id` header is not a valid UUID.
- Returns 200 with a `SessionResponse` for a new avatar (auto-bootstraps).
- Returns 200 with a `SessionResponse` for an existing avatar.
- `level_up_available` is `False` on a fresh avatar.
- `quips` contains at least one entry (the LOGIN quip).
- `events` is a list (may be empty for a fresh avatar).
- The snapshot is saved to the store after a successful open.
- `last_tick` in the stored snapshot equals the injected `now`.
- `sustained_since` is set when `combined_health >= threshold` at session open.
- `sustained_since` is `None` when `combined_health < threshold` at session open.

**`GET /avatar/{id}`**

- Returns 404 for an unknown avatar ID.
- Returns 200 with `AvatarResponse` for a known avatar.
- State is not modified (store snapshot unchanged after GET).
- Returns 422 for a malformed UUID path parameter.

### `tests/api/test_action.py`

- Returns 422 for unknown action kind.
- `POST /action/feed` returns 200 with `ActionResponse` containing a quip.
- `POST /action/clean` returns 200.
- `POST /action/visit` returns 200.
- `POST /action/gift` returns 200.
- `POST /action/chat` returns 200.
- Each action saves the updated snapshot to the store.
- Feed increases hunger in the stored avatar (verify via follow-up
  `GET /avatar/{id}`).
- Clean increases hygiene.
- Visit / gift / chat increase social score.
- `sustained_since` is updated correctly after an action that pushes health
  above threshold.
- `sustained_since` is cleared after an action that drops health below threshold.
- Action on an unknown avatar ID bootstraps and returns 200 (auto-create).

### `tests/api/test_level_up.py`

- Returns 404 for unknown avatar.
- Returns 409 when level-up is not available (fresh avatar, `sustained_since=None`).
- Returns 409 when level-up is not available (`sustained_since` set but not

- Returns 200 with `LevelUpResponse` when level-up is available.
- Returns 422 for invalid `skin_slug`.
- Returns 422 for invalid `room_slug`.
- After a successful level-up, avatar level in the store is incremented.
- After a successful level-up, `sustained_since` in the store is `None`.
- `skin_options` and `room_options` in the response reflect the new level.
- Request body without `skin_slug` returns 422.
- Request body without `room_slug` returns 422.

## Key Design Decisions

1. **No business logic in route handlers** — handlers orchestrate (load → call
   → update → save → respond).  All decisions live in engine modules or `_helpers`.

2. **`make_snapshot` bootstraps on first open, not on `GET /avatar`** — `GET`
   returns 404 for unknown avatars; only a session open creates a new avatar.
   This prevents ghost avatars from being created by speculative reads.

3. **SSE generator body is `# pragma: no cover`** — testing an async generator
   over a live HTTP stream requires infrastructure out of scope for Phase 8.
   The `EventBus` is fully unit-tested; the wiring is integration-tested in Phase 9.

4. **`quip_caller` is per-request, not global** — each request gets a fresh
   unseeded `Random`, producing non-deterministic quips in production.  Tests
   override this dependency with a seeded caller.

5. **`EventBus` publishes game-loop events only** — feed/clean/interact don't
   push events; only `open_session` results (which include tick events) publish
   to the bus.  Action endpoints save the avatar without going through
   `open_session`, so decay events only fire on session open, not inline.

6. **`pace` is stored in `app.state`** — a single instance for the process.
   Production uses `PRODUCTION_PACE`; tests override via `app.state.pace`.
   There is no per-request or per-avatar pace configuration in Phase 8.

7. **Catalogs are module-level constants, not injected** — `SKIN_CATALOG` and
   `ROOM_CATALOG` are imported directly in `level_up.py`.  They are pure
   static data; injecting them would be unnecessary indirection.
