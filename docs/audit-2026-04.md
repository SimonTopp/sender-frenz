## 1. Overall Summary

**Project.** `sender-frenz` is a persistent, multiplayer creature-raising game layered on a
social network. Players raise avatars from bare skeletons to thriving creatures through daily
check-ins spanning physical care (feeding, hygiene) and social engagement (visiting, gifting,
chatting). Avatars visually drift toward vampiric corruption when socially neglected.

**Scale.**

|------|------:|
| Python source files (`src/sender_frenz/`) | 41 |
| Python source LOC | ~4,356 |
| Test files | 47 (35 with test functions) |

| Documentation (`docs/*.md` + root) | 13 markdown files |

**Architecture.** Strict modular dependency hierarchy enforced by `AGENTS.md:94-101`:

```
character_builder  ──┐

required_maintenance ├──►
social_maintenance   ──┘
                         │
                      game_loop (orchestration)

                      api (FastAPI, HTTP, SSE)
                         │
                      web (PWA frontend)
```

Actions workflows (`ci.yml`, `lighthouse.yml`).

**Quality gates (CI-enforced).** `ruff format --check`, `ruff check`, `mypy` strict, `pytest`

**Overall verdict.** The codebase is in excellent shape. It is small, well-scoped, strictly

## 2. Suggested Refactors / Re-architects

Nothing here is urgent. The codebase does not carry architectural debt; these are polish items.

### 2.1 Consolidate duplicated threshold constants — **Low priority**

`HUNGER_IDEAL_MAX` and `HYGIENE_IDEAL_MAX` are defined twice with intentional mirrors:

- `src/sender_frenz/required_maintenance/actions.py:58-62` (source of truth for decay penalties)
- `src/sender_frenz/character_builder/appearance.py:51-75` (mirrored for visual bucketing)

The duplication exists to honor the "no cross-module imports except from `common`" rule
(`AGENTS.md:94-101`). The right fix is to promote these to `common/` — they are universal
game constants, not domain logic, so the architectural rule is not violated by hoisting them.

**Suggested change.** Add a `Thresholds` dataclass (or module-level constants) in
`common/models.py` or a new `common/thresholds.py`; import from both current sites. Drops
~25 lines of mirrored doc-comment maintenance.

**Effort.** Low (mechanical). **Risk.** Low (mypy + 100% coverage will catch drift immediately).

### 2.2 Replace assertion with explicit exception — **Trivial, recommended**

`src/sender_frenz/game_loop/scheduler.py:34`:

```python
snapshot = store.load(avatar_id)
assert snapshot is not None
```

Assertions are stripped under `python -O`. If `list_ids()` ever returns an id `load()` can't
resolve (store bug, race), production would silently proceed with `None`. Prefer:

```python
snapshot = store.load(avatar_id)
if snapshot is None:
    raise ValueError(f"Avatar {avatar_id} listed but not found in store")
```

**Effort.** One line. **Risk.** None (stricter, not laxer).

### 2.3 Persistent store backend — **Planned, not refactor**

`persistence/store.py` ships only `MemoryStore`. On restart, all avatar state is lost. This is
acknowledged in `docs/deploy.md:15` and is on the roadmap (`docs/platform-roadmap.md`). The
`StoreProtocol` design is correct — a SQLite/Postgres implementation can drop in without
touching callers. **No action required now.**

### 2.4 SSE coverage exclusion — **Informational**

`src/sender_frenz/api/routes/stream.py:36-67` carries `# pragma: no cover` with a note
deferring live-stream testing to Phase 9. Phase 9 is in progress. If SSE integration tests
don't land this phase, formalize the exclusion with a pytest marker rather than a pragma.

### 2.5 Things that look like smells but are intentional

- **`character_builder/skins.py` catalog length.** A flat catalog of 12 skins × 4 tiers reads
  as a large file but it is declarative data, not logic. Leave it alone.
- **Each module's own threshold constants beyond 2.1.** Module-local tunables (decay rates,
  interaction weights) *should* live with their owning module.
- **No shared base class across actions.** Each action is a pure function. The absence of a
  `BaseAction` abstraction is correct for ~400 LOC of simple, pure logic.

## 4. Agent Accessibility

**Verdict: excellent.** This repo is among the more agent-friendly codebases I've audited.

### 4.1 `AGENTS.md` is load-bearing and current

15 KB / 400 lines, covering: philosophy, plan lifecycle, self-documenting practices, tooling,
coverage mandate, doc-freshness rule, pacing rules, core contracts (`sustained_since`,
`GameEvent`, `TYPE_CHECKING`), AI-first practices, aesthetic gate, pre-commit workflow, and
commit conventions. Line 108-110 explicitly names it authoritative.

No `CLAUDE.md` exists, and none is needed — `AGENTS.md` already plays that role.

### 4.2 Commands, entry points, and discoverability

All quality-gate commands are documented and working:

| Command | Documented |
|---------|------------|
| `uv sync --group dev` | `AGENTS.md:130` |
| `uv run ruff format --check .` | `README.md:131` |
| `uv run ruff check .` | `README.md:132` |
| `uv run mypy .` | `README.md:133` |
| `uv run pytest` | `README.md:134` |
| `uv run uvicorn sender_frenz.api:app` | `api/app.py` docstring |
| `uv run pytest tests/pwa` | `tests/pwa/conftest.py:10` |

Entry points are unambiguous: `src/sender_frenz/api/app.py` for the backend, `web/index.html`
for the frontend, `pyproject.toml` for all tool config.

### 4.3 Structural guardrails that help agents

- **Import hierarchy is enforced.** "No cross-module imports except from `common`" eliminates
  an entire class of "where should this live?" confusion.
- **Pure functions.** Actions, decay, and tick are side-effect-free, so tests read as truth
  tables and agents can reason locally.
- **Strict types.** `mypy --strict` catches contract drift before CI.
- **Centralized config.** All tool settings in `pyproject.toml`; no `setup.cfg`/`.flake8`/`tox.ini`
  scavenger hunt.

### 4.4 Friction points (minor)

- Agents must read `docs/aesthetic.md` before writing player-facing strings
  (`AGENTS.md:335`). Correct gate, but it's an extra step.
- The `GamePace` injection pattern (`AGENTS.md:194-220`) must be understood before any
  time-based change. Well-documented, but non-trivial.
- Plan docs cross-reference each other; new agents may need to traverse 2-3 docs to get full
  context for a given module.

### 4.5 Onboarding walkthrough

A new agent can go from zero to productive in ~30 minutes: read `AGENTS.md`, skim `README.md`,
read the relevant `docs/<module>.md`, run the quality gate, and start editing. 35 existing
test files provide concrete templates.
